# MCP 远程授权：从 401 到第一次受保护调用

远程 MCP Server 通过网络接收请求，不能因为 Client 知道 `/mcp` 地址，就默认允许它
调用 Tool。

本文是一个跟做型实验。读完并跑完以后，你应该能回答：

> 同一个 `/mcp` endpoint，为什么不带 Token 的请求会被拒绝，而带着授权服务签发的
> access token 的请求可以进入 MCP 会话并调用 Tool？

这个实验实际验证两条链路：

```text
链路 A：无 Token 对照
Client 不带 Token 请求 /mcp
→ Resource Server 在认证层返回 HTTP 401

链路 B：带 Token 的受保护调用
Client 主动向 Authorization Server 发起授权请求
→ 用户登录并同意，得到一次性 authorization code
→ Client 用 code 换 access token
→ Client 携带 access token 请求 /mcp
→ Resource Server introspection 校验 Token
→ MCP initialize 成功
→ get_order Tool 执行成功
```

第一条链路只用来证明 `/mcp` 入口确实受保护；第二条链路演示 Client 如何主动取得 Token，并发起一次带 Token 的受保护调用。

注意，这不是生产级 OAuth 实现。它没有真实浏览器登录页、PKCE、动态 Client 注册、刷新
Token、撤销接口和持久化存储。它的目标更窄：让你看清远程 MCP 授权中最关键的几个
动作，以及这些动作分别发生在哪一层。

## 1. 实验文件与角色

实验使用三个文件：

| 文件 | 角色 | 作用 |
| --- | --- | --- |
| `examples/remote_auth_server.py` | Authorization Server | 模拟登录、同意、签发 authorization code、签发 access token，并提供 introspection |
| `examples/remote_auth_resource_server.py` | MCP Resource Server | 暴露受保护的 `/mcp` endpoint，验证 Bearer Token 后才允许调用订单 Tool |
| `examples/remote_auth_client.py` | MCP Client / OAuth Client | 先不带 Token 观察 401，再走授权码流程取得 access token，最后调用 Tool |

这三个角色不要混在一起：

- Authorization Server 负责发凭据，不处理 MCP Tool；
- Resource Server 负责保护 MCP Tool，不负责用户登录；
- Client 先向授权服务拿 token，再拿 token 访问 MCP Server。

示例里固定了演示账号和应用身份：

```text
用户账号：reader@example.com / letmein
Client AccessKey：ai-forge-demo-client
Client SecretKey：ai-forge-demo-secret
权限范围：orders:read
```

这里的 `CLIENT_ACCESS_KEY` 和 `CLIENT_SECRET_KEY` 会放入 OAuth 表单字段
`client_id` 和 `client_secret`。它们代表“哪个应用在申请授权”，不是最终访问 MCP
Server 的凭据。真正访问 MCP Server 的是运行时签发出来的 `access_token`。

## 2. 先看两条对照数据流

运行前先把实验的两个场景放在脑子里。它们在同一个 Client 脚本里连续执行，但逻辑上是
两条对照链路。

```text
链路 A：无 Token 请求

remote_auth_client.py
  1. POST /mcp，不带 Authorization Header
       ↓
remote_auth_resource_server.py
  2. FastMCP 认证层拒绝请求，返回 401

链路 B：带 Token 请求

remote_auth_client.py
  3. GET /authorize，发起授权请求
       ↓
remote_auth_server.py
  4. 检查 client_id 和 scope，返回教学提示

remote_auth_client.py
  5. POST /authorize/approve，提交演示用户、密码和 consent=yes
       ↓
remote_auth_server.py
  6. 生成一次性 authorization code，保存到 authorization_codes

remote_auth_client.py
  7. POST /token，提交 code、redirect_uri、client_id、client_secret
       ↓
remote_auth_server.py
  8. pop 掉 code，确认只能用一次，签发 access token

remote_auth_client.py
  9. Authorization: Bearer <access_token> 请求 /mcp
       ↓
remote_auth_resource_server.py
 10. FastMCP 认证层调用 DemoTokenVerifier.verify_token()
       ↓
remote_auth_server.py
 11. /introspect 返回 token 是否 active、iss、aud、scope、exp
       ↓
remote_auth_resource_server.py
 12. 校验通过，MCP initialize 和 get_order 才会执行
```

下面按这个顺序看代码和运行结果。

## 3. Authorization Server 做了什么

`remote_auth_server.py` 是教学用授权服务，默认监听：

```text
http://127.0.0.1:9000
```

它用两个内存字典保存状态：

```python
authorization_codes: dict[str, dict[str, object]] = {}
access_tokens: dict[str, dict[str, object]] = {}
```

重启服务后，授权码和 Token 都会丢失。这正好适合实验：每次运行都能看到
access token 是现场签发出来的，不是写死在 Client 里的固定字符串。

### `/authorize`：发起授权请求

Client 第一步访问 `/authorize`：

```python
hint = await client.get(
    f"{AUTH_SERVER_URL}/authorize",
    params={
        "client_id": CLIENT_ACCESS_KEY,
        "redirect_uri": REDIRECT_URI,
        "scope": SCOPE,
        "state": "ai-forge-demo",
    },
)
```

这一步的作用不是签发 Token。它表示：

```text
我是 ai-forge-demo-client
我想申请 orders:read
用户同意后请回到 http://127.0.0.1:8765/callback
```

真实 OAuth 中，这一步通常会打开浏览器并展示登录/授权页面。本实验不引入页面，所以
`/authorize` 只检查 `client_id` 和 `scope`，然后返回一段提示。

### `/authorize/approve`：用户登录并同意

Client 接着模拟用户输入账号密码并同意授权：

```python
approve = await client.post(
    f"{AUTH_SERVER_URL}/authorize/approve",
    data={
        "client_id": CLIENT_ACCESS_KEY,
        "redirect_uri": REDIRECT_URI,
        "scope": SCOPE,
        "state": "ai-forge-demo",
        "username": DEMO_USER,
        "password": DEMO_PASSWORD,
        "consent": "yes",
    },
)
code = approve.json()["code"]
```

Authorization Server 会生成一次性 `authorization code`：

```python
code = token_urlsafe(24)
authorization_codes[code] = {
    "client_id": client_id,
    "redirect_uri": redirect_uri,
    "username": username,
    "scope": scope,
    "expires_at": time() + CODE_TTL_SECONDS,
}
```

`code` 的作用是证明“用户刚刚同意过这次授权请求”。它不是访问凭据，不能拿去调用
MCP Server。它只是用来换 access token 的一次性凭证。

### `/token`：用 code 换 access token

Client 再把 `code` 提交给 `/token`：

```python
token = await client.post(
    f"{AUTH_SERVER_URL}/token",
    data={
        "grant_type": "authorization_code",
        "code": code,
        "redirect_uri": REDIRECT_URI,
        "client_id": CLIENT_ACCESS_KEY,
        "client_secret": CLIENT_SECRET_KEY,
    },
)
access_token = token.json()["access_token"]
```

这里同时提交几类信息：

- `code`：证明用户刚刚同意过；
- `redirect_uri`：证明这次换票和前面的授权请求属于同一条链路；
- `client_id` / `client_secret`：证明是这个 Client 在换票。

服务端用这一句取出授权码：

```python
code_record = authorization_codes.pop(code, None)
```

`pop` 的意思是：如果字典里有这个 `code`，取出它对应的记录，同时把它删除；如果没有，
返回 `None`。这样 authorization code 只能使用一次。第一次换 token 成功以后，第二次
再拿同一个 code 来换，就会得到 `invalid_code`。

最后，Authorization Server 生成真正能访问 MCP Server 的 `access_token`：

```python
access_tokens[access_token] = {
    "client_id": client_id,
    "username": code_record["username"],
    "scope": code_record["scope"],
    "aud": MCP_RESOURCE,
    "iss": ISSUER_URL,
    "expires_at": time() + TOKEN_TTL_SECONDS,
}
```

这里的几个字段后面会被 Resource Server 校验：

| 字段 | 含义 |
| --- | --- |
| `scope` | 这个 Token 有什么权限，本实验需要 `orders:read` |
| `aud` | 这个 Token 是签给哪个 Resource Server 用的 |
| `iss` | 这个 Token 是哪个 Authorization Server 签发的 |
| `expires_at` | Token 什么时候过期 |

## 4. Resource Server 如何挡住未授权请求

`remote_auth_resource_server.py` 是真正的远程 MCP Server，默认暴露：

```text
http://127.0.0.1:8001/mcp
```

它提供一个订单查询 Tool：

```python
@mcp.tool()
def get_order(order_id: OrderId) -> OrderFound | OrderNotFound:
    ...
```

注意：`get_order()` 里没有手动调用 `verify_token()`。认证发生在 Tool 执行之前。

Resource Server 把 `DemoTokenVerifier()` 注册给 `FastMCP`：

```python
mcp = FastMCP(
    "shop-order-remote-auth",
    ...,
    token_verifier=DemoTokenVerifier(),
    auth=AuthSettings(
        issuer_url=AnyHttpUrl(AUTH_SERVER_URL),
        resource_server_url=AnyHttpUrl(MCP_URL),
        required_scopes=[REQUIRED_SCOPE],
    ),
)
```

当运行：

```python
mcp.run(transport="streamable-http")
```

FastMCP 会创建 Streamable HTTP app，并在 `/mcp` 入口外面挂认证层。请求链路可以理解为：

```text
HTTP 请求进入 /mcp
→ 认证层读取 Authorization: Bearer <token>
→ 调用 DemoTokenVerifier.verify_token(token)
→ 检查返回的 AccessToken 是否包含 orders:read
→ 通过后才进入 MCP initialize / tools-call
→ 最后才可能执行 get_order()
```

所以错误 Token 不是在 `get_order()` 里失败，而是在进入 MCP Tool 之前就被拒绝。

`DemoTokenVerifier.verify_token()` 的核心逻辑是调用授权服务的 `/introspect`：

```python
response = await client.post(
    f"{AUTH_SERVER_URL}/introspect",
    data={"token": token},
)
```

然后检查四件事：

```python
if (
    not token_info.get("active")
    or token_info.get("iss") != AUTH_SERVER_URL
    or token_info.get("aud") != MCP_URL
    or REQUIRED_SCOPE not in scopes
):
    return None
```

这四项分别表示：

| 校验项 | 作用 |
| --- | --- |
| `active` | Token 是否仍然有效 |
| `iss` | Token 是否由当前 Authorization Server 签发 |
| `aud` | Token 是否签给当前 MCP Resource Server 使用 |
| `scope` | Token 是否包含调用订单查询所需的 `orders:read` |

如果任何一项失败，`verify_token()` 返回 `None`，FastMCP 认证层会拒绝请求。只有通过后，
它才返回 `AccessToken`：

```python
return AccessToken(
    token=token,
    client_id=token_info["client_id"],
    scopes=scopes,
)
```

这个返回值告诉 FastMCP：请求已经认证通过，并且拥有这些 scope。

## 5. Client 如何形成对照

`remote_auth_client.py` 先故意不带 Token 请求 `/mcp`：

```python
async with httpx.AsyncClient() as client:
    response = await client.post(MCP_URL, json={})
print(f"无 Token：HTTP {response.status_code}")
```

这不是合法的 MCP 初始化消息，但认证发生在 MCP 消息处理之前，所以足以观察入口是否
被保护。预期输出是：

```text
无 Token：HTTP 401
```

然后 Client 执行三步授权流程：

```text
GET  /authorize
POST /authorize/approve
POST /token
```

拿到 `access_token` 后，再把它放进 HTTP Header：

```python
async with httpx.AsyncClient(
    headers={"Authorization": f"Bearer {access_token}"}
) as authenticated_client:
    async with streamable_http_client(
        MCP_URL,
        http_client=authenticated_client,
    ) as (read, write, _):
        async with ClientSession(read, write) as session:
            initialized = await session.initialize()
            result = await session.call_tool(
                "get_order",
                {"order_id": "O-1001"},
            )
```

这一次请求会先经过 Resource Server 的认证层。Token 通过 introspection 校验以后，
Client 才能建立 MCP Session，并发送 `initialize` 和 `tools/call`。

## 6. 运行实验

第一次运行前，在仓库根目录安装依赖：

```bash
uv sync
```

打开三个终端，按顺序启动。

终端一启动 Authorization Server：

```bash
uv run labs/mcp/foundations/examples/remote_auth_server.py
```

它会监听：

```text
http://127.0.0.1:9000
```

终端二启动受保护的 MCP Resource Server：

```bash
uv run labs/mcp/foundations/examples/remote_auth_resource_server.py
```

它会监听：

```text
http://127.0.0.1:8001/mcp
```

终端三运行 Client：

```bash
uv run labs/mcp/foundations/examples/remote_auth_client.py
```

Client 关键输出应包含：

```text
无 Token：HTTP 401
1. 发起授权请求
   授权服务：用演示账号登录并同意授权后，POST /authorize/approve 会返回授权码。
   申请权限：orders:read
2. 用户登录并同意
   已取得一次性 authorization code
3. 授权码换 Token
   Token 端点已签发短期 access token
正确 Token：协议版本 2025-11-25
受保护 Tool：{'result': {'found': True, 'order': {
    'order_id': 'O-1001',
    ...
}}}
```

这几段输出分别证明：

- 不带 Token 的请求在 `/mcp` 入口被拒绝；
- Client 发起了授权请求，并申请 `orders:read`；
- 用户登录和同意后，Authorization Server 签发了 authorization code；
- Client 用 code 换到了 access token；
- 携带正确 access token 后，MCP `initialize` 和 `get_order` 都执行成功。


## 7. 常见困惑

**`/authorize` 有什么用？**

它表示“发起授权请求”：Client 告诉 Authorization Server 自己是谁、要什么权限、用户
同意后回哪里。真实系统通常在这里展示登录/授权页面。本实验为了简单，只返回提示信息。

**authorization code 和 access token 有什么区别？**

authorization code 是一次性换票凭证，不能访问 MCP Server。access token 是访问凭据，
会放进：

```text
Authorization: Bearer <access_token>
```

Resource Server 只接受 access token，不接受 authorization code。

**为什么 `get_order()` 里看不到认证代码？**

因为认证发生在 FastMCP 的 HTTP 入口层。`token_verifier=DemoTokenVerifier()` 会被
FastMCP 接入认证中间件。只有认证通过，请求才会进入 MCP `tools/call`，最终执行
`get_order()`。

**为什么要 introspection？**

本实验使用 opaque token，也就是 Resource Server 不能只靠本地解析 token 判断有效性。
它需要把 token 发给 Authorization Server 的 `/introspect`，询问这个 token 是否
active、由谁签发、给谁使用、有什么 scope、是否过期。

## 8. 边界与验收

这个实验比固定 Token 更接近真实系统，但仍然不是完整 OAuth：

- 演示账号、密码、Client AccessKey 和 Client SecretKey 写在代码里；
- 登录和同意用 HTTP 表单模拟，没有真实浏览器页面和回调监听；
- 没有 PKCE、动态 Client 注册、刷新 Token 和撤销接口；
- 授权码和 Token 保存在内存里，服务重启后丢失；
- 没有演示 Client 如何从 `401` 自动发现授权服务。

本阶段最终要记住的是：

> 远程 MCP 授权不是 Tool 里的一个 `if` 判断，而是 HTTP/MCP 入口前的一道认证门槛：
> 只有 Bearer Token 被验证通过，请求才会进入 MCP Session 和 Tool 执行。
