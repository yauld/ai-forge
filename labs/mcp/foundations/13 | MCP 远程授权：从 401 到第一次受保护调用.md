# MCP 远程授权：从 401 到第一次受保护调用

本地 stdio Server 通常由 Host 直接启动，进程继承了本机用户授予 Host 的环境与
权限。远程 MCP Server 则通过网络同时服务多个用户和 Client，不能因为请求到达
`/mcp` 就默认信任调用者。

这一阶段只回答一个问题：

> 没有凭据的 MCP Client，如何获得远程 MCP Server 接受的 access token，并完成
> 第一次受保护的 Tool 调用？

本文会跑通一条最小授权链路：

```text
无 Token 请求 → 401 → 发现授权服务 → PKCE 授权
             → 授权码换 Token → 携带 Token 调用 Tool
```

完成实验后，你应该能：

- 区分用户、Host、MCP Client、MCP Server、Authorization Server 和业务系统；
- 解释 `401` 如何把 Client 引向授权服务；
- 说明 authorization code 与 access token 分别发给谁；
- 解释 PKCE 和 `resource` 参数在流程中的位置；
- 用真实输出证明未授权请求被拒绝、授权后调用成功。

细粒度 scope、audience 校验和 token passthrough 留到阶段 14，远程攻击面留到
阶段 15。本阶段不把它们挤进第一次授权实验。

## 1. 实验文件与运行前提

实验复用订单查询业务，新增四个文件：

| 文件 | 角色 |
| --- | --- |
| `examples/remote_auth_server.py` | Authorization Server，提供注册、授权、Token 和 introspection endpoint |
| `examples/remote_auth_provider.py` | 保存动态注册 Client、授权码和 Token 的教学用内存状态 |
| `examples/remote_auth_resource_server.py` | 受保护的远程 MCP Server，也就是 OAuth Resource Server |
| `examples/remote_auth_client.py` | MCP Client，执行发现、PKCE 授权和订单查询 |

仓库要求 Python 3.13 或更高版本，依赖由 `pyproject.toml` 管理。第一次运行时在
仓库根目录执行：

```bash
uv sync
```

实验使用两个本地地址：

| 地址 | 用途 |
| --- | --- |
| `http://127.0.0.1:9000` | Authorization Server |
| `http://127.0.0.1:8001/mcp` | 受保护的 MCP endpoint |

生产环境中的 OAuth endpoint 应使用 HTTPS；localhost HTTP 只用于本地开发实验。

## 2. 先分清六个角色

授权链路比 stdio 多出一个 Authorization Server，但 MCP 的 Host、Client、Server
职责没有改变：

```text
用户
  ↓ 批准访问
Host：remote_auth_client.py
  ├─ MCP Client：发送 MCP 消息
  └─ OAuth Client：发现授权服务、申请并保存 Token
         ↓
Authorization Server：注册 Client、确认授权、签发 Token
         ↓
远程 MCP Server / Resource Server：验证 Token、处理 tools/call
         ↓
订单数据库：返回 O-1001
```

这里容易混淆两组名称：

- **MCP Client** 描述谁与 MCP Server 交换 MCP 消息；
- **OAuth Client** 描述谁代表用户向 Authorization Server 申请访问凭据。

本实验中它们都位于 `remote_auth_client.py`，但属于两套协议中的不同角色。

远程 MCP Server 是 OAuth Resource Server：它保护订单 Tool，但不负责让用户
登录。Authorization Server 负责用户授权和 Token 签发。MCP 当前授权规范也采用
这种分离方式。

官方参考：
[MCP Authorization Specification](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)

## 3. 第一步：未授权请求必须得到 401

实验先用普通 HTTP Client，故意不携带 Token 请求 `/mcp`：

```python
async with httpx.AsyncClient() as http_client:
    response = await http_client.post(MCP_URL, json={})
```

这里的 `http_client` 不是 MCP Client。它只负责发送普通 HTTP 请求；真正的 MCP
Client 是后文基于 `ClientSession` 建立的会话。这次请求也不是合法的 MCP 初始化
消息，但它会先经过认证中间件，所以能够单独观察授权边界。预期结果是：

```text
1. no token -> HTTP 401
   WWW-Authenticate: Bearer ... resource_metadata="http://127.0.0.1:8001/.well-known/oauth-protected-resource/mcp"
```

`401` 不只表示“失败”。`WWW-Authenticate` 中的 `resource_metadata` 还告诉
Client：到哪里读取这个受保护资源的元数据。

MCP Server 由 `AuthSettings` 声明自己的两个关键地址：

```python
auth=AuthSettings(
    issuer_url=AnyHttpUrl("http://127.0.0.1:9000"),
    resource_server_url=AnyHttpUrl("http://127.0.0.1:8001/mcp"),
    required_scopes=["orders:read"],
)
```

其中：

- `resource_server_url` 标识被保护的 MCP Server；
- `issuer_url` 指向可以为它签发 Token 的 Authorization Server。

## 4. 第二步：从 Resource Server 发现 Authorization Server

Client 根据 `resource_metadata` 获取 Protected Resource Metadata：

```json
{
  "resource": "http://127.0.0.1:8001/mcp",
  "authorization_servers": [
    "http://127.0.0.1:9000/"
  ],
  "scopes_supported": [
    "orders:read"
  ]
}
```

实验输出只保留两个关键字段：

```text
2. protected resource: http://127.0.0.1:8001/mcp
   authorization servers: ['http://127.0.0.1:9000/']
```

接着，SDK 会从 Authorization Server 的 metadata endpoint 发现：

- 动态 Client 注册 endpoint；
- authorization endpoint；
- token endpoint；
- 支持的 PKCE 方法。

因此 Client 不需要把 `/authorize`、`/register`、`/token` 等路径逐个写死在业务
调用逻辑中。发现过程把“我要访问哪个 Resource Server”和“谁能为它签发 Token”
连接起来。

## 5. 第三步：用 PKCE 取得授权码

`OAuthClientProvider` 先动态注册本次实验 Client，再创建授权请求。Client metadata
声明了回调地址和所需 scope：

```python
OAuthClientMetadata(
    client_name="AI Forge Stage 12 Client",
    redirect_uris=[AnyUrl("http://127.0.0.1:3030/callback")],
    grant_types=["authorization_code", "refresh_token"],
    response_types=["code"],
    scope="orders:read",
)
```

授权请求包含两个本阶段需要观察的字段：

```text
code_challenge_method=S256
resource=http://127.0.0.1:8001/mcp
```

PKCE 的基本过程是：

1. Client 本地生成随机 `code_verifier`；
2. Client 把它计算成 `code_challenge`，放进授权请求；
3. Authorization Server 把 challenge 与授权码绑定；
4. Client 换 Token 时提交原始 verifier；
5. Authorization Server 验证二者是否匹配。

这样，即使授权码在回调途中被截获，没有原始 verifier 的一方也不能直接拿它换
Token。实验不打印 verifier、授权码或完整 Token，因为凭据不应该进入普通日志。

`resource` 则明确告诉 Authorization Server：Client 申请的 Token 将用于哪个
MCP Resource Server。阶段 14 会继续验证 Server 如何使用这个边界。

真实桌面 Client 通常会打开浏览器，让用户完成登录和同意。本实验的
`create_automated_user_agent()` 会自动访问本地 `/approve` 页面，模拟用户点击
“允许”，使实验可以重复运行而不需要复制 callback URL。它没有绕过 PKCE：
授权码交换和 verifier 校验仍由 SDK 的 OAuth 路由完成。

## 6. 第四步：授权码换取 access token

用户批准后，Authorization Server 把浏览器重定向到 Client 注册的 callback：

```text
http://127.0.0.1:3030/callback?code=...&state=...
```

这里返回的是短期、一次性使用的 authorization code，不是 access token。
`OAuthClientProvider` 会在后台把以下信息发送给 token endpoint：

- authorization code；
- PKCE `code_verifier`；
- Client 注册信息；
- 目标 `resource`。

验证通过后，Authorization Server 才向 OAuth Client 返回 access token。
`InMemoryTokenStorage` 保存 Token，后续 HTTP 请求由 SDK 自动添加：

```text
Authorization: Bearer <access-token>
```

为了证明 Token 已经取得，同时避免泄露完整值，Client 只输出短前缀：

```text
4. access token: token_niKwUw... (redacted)
```

本实验的 Client 和 Authorization Server 都只在内存中保存状态，进程退出后立即
丢失。这适合可重复教学，不代表生产凭据存储方案。真实 Host 应使用操作系统安全
存储或受控密钥存储，并正确处理过期、刷新和撤销。

## 7. 第五步：携带 Token 调用受保护 Tool

Resource Server 不直接读取 Authorization Server 的内存。它收到 Bearer Token
后调用 `/introspect`，确认 Token 仍然有效并取得 Client、scope、过期时间等信息。
验证通过后，请求才进入 MCP Session：

```python
initialized = await session.initialize()
result = await session.call_tool("get_order", {"order_id": "O-1001"})
```

完整实验的最后一段输出是：

```text
5. protocol: 2025-11-25
   protected tool result: {'result': {'found': True, 'order': {
       'order_id': 'O-1001',
       'order_date': '2026-06-19',
       'status': 'paid',
       'amount': 199.0,
       'region': '华东',
       'product': '耳机'
   }}}
```

这段结果与最开始的 `401` 形成对照：

- 同一个 MCP endpoint，没有 Token 时在认证层被拒绝；
- 完成授权后，Client 可以初始化 MCP Session 并调用订单 Tool；
- MCP Tool 本身没有负责登录或签发 Token；
- Authorization Server 也没有执行订单查询。

## 8. 运行完整实验

打开三个终端，都从仓库根目录执行。

终端一启动 Authorization Server：

```bash
uv run labs/mcp/foundations/examples/remote_auth_server.py
```

终端二启动受保护的 MCP Resource Server：

```bash
uv run labs/mcp/foundations/examples/remote_auth_resource_server.py
```

终端三运行 Client：

```bash
uv run labs/mcp/foundations/examples/remote_auth_client.py
```

一次通过的关键输出应包含：

```text
1. no token -> HTTP 401
2. protected resource: http://127.0.0.1:8001/mcp
3. authorization request includes PKCE and resource
4. access token: ... (redacted)
5. protocol: 2025-11-25
   protected tool result: ...
```

如果端口已经被占用，可以先停止占用 `8001` 或 `9000` 的旧进程。Client 必须在
两个 Server 都启动后运行；Resource Server 需要访问 Authorization Server 的
introspection endpoint。

## 9. 这个实验能证明什么

实验动态证明了：

- MCP Server 会拒绝没有 Token 的 HTTP 请求；
- `401` 提供了 Protected Resource Metadata 地址；
- Client 能发现独立的 Authorization Server；
- 授权请求使用 PKCE，并声明目标 `resource`；
- Authorization Server 签发 access token 后，受保护 Tool 可以成功调用。

实验没有证明：

- 当前内存 Authorization Server 适合生产；
- Token 已经使用签名 JWT 或经过完整 audience 策略校验；
- `orders:read` 已经和退款等危险能力分开；
- refresh token、撤销和安全持久化已经实现；
- confused deputy、SSRF 与 session hijacking 已经得到防护。

这些边界不是遗漏。阶段 13 的目标是先看清正常授权路径；阶段 14 再研究“拿到
Token 后能做什么”，阶段 15 再主动攻击远程授权和 Session 边界。
