# MCP 审计安全：既要留下证据，也不能泄露秘密

危险操作成功或失败后，系统需要留下调查证据。但如果审计直接保存完整 Tool
请求，幂等键、Token 或业务敏感字段又可能随着日志泄露。

本文只回答一个问题：

> MCP Server 如何保留足够的操作证据，同时避免审计日志成为新的敏感信息泄露面？

## 1. 实验文件与范围

| 文件 | 作用 |
| --- | --- |
| `examples/audit_security_server.py` | 执行退款并写入最小化审计记录 |
| `examples/audit_security_client.py` | 提交成功与失败请求，检查审计和最终状态 |
| `examples/security_order_data.py` | 重置独立的样例订单数据库 |

首次运行前执行：

```bash
uv sync
```

本实验不调用模型。研究对象是请求到达 Server 后，业务执行与审计写入之间的
边界。

## 2. 审计数据如何产生

实验的数据流是：

```text
Client 调用 refund_order
  ↓
Server 根据订单状态执行或拒绝
  ↓
只选择调查需要的字段
  ↓
原始幂等键转换为 SHA-256 短指纹
  ↓
写入 audit_events
  ↓
Client 读取审计并回查订单最终状态
```

审计记录保留：

- 时间、动作、订单编号和结果；
- 成功退款的金额和业务原因；
- 原始幂等键摘要的前 12 位短指纹。

审计不保存：

- 原始幂等键；
- 完整 Tool 请求；
- 与调查无关的订单字段。

短指纹可以关联同一业务意图，但不能还原原始幂等键。

## 3. 为什么同时测试成功与拒绝

实验提交两笔请求：

| 请求 | 预期结果 | 审计价值 |
| --- | --- | --- |
| 对已支付订单 O-1005 退款 | `applied` | 证明成功副作用可以追踪 |
| 对不存在订单 O-9999 退款 | `denied` | 证明失败尝试同样留下证据 |

只记录成功操作会遗漏攻击尝试，只记录失败又无法解释真实业务状态变化。两种结果
都应可调查，但都不应泄露原始凭证。

## 4. 运行实验

在仓库根目录执行：

```bash
uv run labs/mcp/foundations/examples/audit_security_client.py
```

预期核心检查：

```text
合法退款成功，非法对象被拒绝                  passed=True
审计同时保留 applied 与 denied               passed=True
审计没有保存原始幂等键                        passed=True
审计保留可关联的幂等键短指纹                  passed=True
成功审计对应的订单最终为 refunded             passed=True
```

最后一项把审计证据与业务状态对应起来。只看一条 `applied` 日志，不能证明订单
真的完成了退款。

## 5. 证据边界

本实验直接观察到：

- 成功和拒绝事件都进入审计；
- 两个原始幂等键都没有出现在审计返回结果中；
- 两条记录都保留了短指纹；
- O-1005 的最终状态为 `refunded`。

这些证据支持“审计可调查且经过数据最小化”，但短指纹仍属于关联标识，不应公开
暴露或无限期保留。真实系统还需要结合访问控制、保留期限、日志完整性和告警策略。

## 6. 常见问题与验收

| 现象 | 优先检查 |
| --- | --- |
| 审计出现 `refund-audit0001` | 是否把完整请求或原始幂等键写入 `detail` |
| 没有 `denied` 事件 | 拒绝分支是否调用 `record_audit()` |
| 没有短指纹 | 是否在写日志前计算摘要 |
| O-1005 不是 `refunded` | 业务更新与审计写入是否在同一事务中完成 |

完成后应能解释：

1. 为什么审计不能直接保存完整 Tool 请求？
2. 为什么失败操作也需要审计？
3. 短指纹解决了什么问题，又没有解决什么问题？
4. 为什么审计结果还要与最终业务状态相互验证？

最终结论是：

```text
审计要留下足够证据
  → 只记录调查所需字段
  → 敏感值使用不可逆摘要关联
  → 再用最终业务状态验证日志含义
```
