# 01 | MCP 是什么：先把它放回 AI 应用架构里理解

如果你刚开始学习 MCP，最容易踩的坑不是语法，而是把它放错位置。

很多介绍会说 MCP 能让 AI 使用工具、读取文件、查询数据库。这个说法没有完全错，但如果只停在这里，很容易误以为 MCP 是一个工具平台，或者是让大模型直接连接外部系统的魔法接口。

更准确地说：

> MCP 是一种开放协议，用来让 AI 应用以标准方式连接外部系统。

它不是大模型，不是 Agent 框架，也不是某一个具体工具。它定义的是 AI 应用和外部系统之间如何交换上下文、发现能力、调用能力。

## 1. MCP 解决的是什么问题

大模型本身擅长理解、推理和生成文本，但它并不能天然访问你的本地文件、公司数据库、GitHub 仓库、订单系统或浏览器环境。

如果一个 AI 应用想使用这些外部能力，通常需要解决几个问题：

- 外部系统有哪些能力可以用？
- 每个能力需要什么输入？
- 调用后返回什么结果？
- 哪些内容只是上下文，哪些能力会执行动作？
- 不同 AI 应用和不同外部系统之间，能不能用统一方式连接？

MCP 要解决的就是这类连接问题。

官方文档把 MCP 类比成 AI 应用的 USB-C 接口：不同设备可以通过统一接口连接，AI 应用也可以通过 MCP 以统一方式连接数据源、工具和工作流。

![MCP 官方示意图：AI 应用通过 MCP 连接外部数据源、工具和工作流](assets/mcp-simple-diagram.avif)

图里最重要的不是某一个具体工具，而是中间这层标准连接方式。左边可以是不同 AI 应用，右边可以是不同数据源、工具或工作流；MCP 的价值在于让两边不必为每一种组合都重新设计一套私有适配。

换句话说，MCP 关注的是“AI 应用如何发现和使用外部能力”。它让 MCP Server 可以代表外部系统暴露能力，让 AI 应用通过 MCP Client 连接这些 Server。真正读取数据、调用 API 或执行动作的，仍然是 Host 和 Server 之间受协议约束的交互过程，而不是大模型自己越过应用直接访问外部系统。

这个类比有助于建立直觉，但也不要过度理解。MCP 标准化的是连接方式，不是替你决定模型如何推理，也不是替你保证所有工具调用都安全。

## 2. MCP 连接的是谁和谁

理解 MCP 时，先不要从“大模型直接调用工具”开始，而应该从 AI 应用架构开始。

一个简化链路是：

```text
用户
  ↓
AI 应用 / MCP Host
  ↓
MCP Client
  ↓
MCP Server
  ↓
外部系统：文件、数据库、API、业务服务、工作流
```

这里有三个核心角色。

**MCP Host**

Host 是使用 MCP 的 AI 应用，例如 Cursor、Visual Studio Code、Claude Desktop、ChatGPT 或某个企业内部 Agent 平台。

Host 负责接收用户请求、调用大模型、管理可用能力，并决定如何把外部上下文和工具结果纳入对话流程。

**MCP Client**

Client 是 Host 内部用来连接某一个 MCP Server 的组件。

一个 Host 可以连接多个 MCP Servers。通常每连接一个 Server，Host 内部就会维护一个对应的 MCP Client。这个 Client 负责和 Server 建立连接、完成初始化、发现能力、发起调用和接收结果。

**MCP Server**

Server 是遵守 MCP 协议、向 Host 暴露能力的程序。

它可以运行在本地，也可以部署在远程。它本身不一定就是数据库、文件系统或 GitHub，而更像是 AI 应用和这些外部系统之间的适配层。

比如在一个 MySQL 查询场景里：

```text
你
  ↓
Cursor
  ↓
Cursor 内部的 MCP Client
  ↓
MySQL MCP Server
  ↓
MySQL 数据库
```

这里 Cursor 是 MCP Host，MySQL MCP Server 是协议适配层，MySQL 数据库才是外部系统。

## 3. 大模型并不是直接连接数据库

一句常见但不严谨的说法是：

> MCP 让大模型直接连接数据库。

更准确的说法应该是：

> MCP 让 AI 应用通过 MCP Client 连接 MCP Server；MCP Server 再把数据库相关能力暴露给 AI 应用。执行 SQL 查询这类动作通常是 tool，数据库 schema、表结构说明这类上下文更像 resource。模型可以基于 Host 提供的上下文生成工具调用意图，但真正执行调用、接收结果、再交回模型组织回答的是 Host。

以“查询今天新增用户数”为例，流程大致是：

```text
1. Cursor 连接 MySQL MCP Server。
2. MySQL MCP Server 告诉 Cursor：我有 query_database 这个 tool。
3. Cursor 把可用 tool 信息纳入模型上下文。
4. 用户问：今天新增了多少用户？
5. 模型判断需要查数据库，并生成 tool call 意图。
6. Cursor 根据权限和策略执行这个 tool call。
7. MySQL MCP Server 查询 MySQL。
8. 查询结果返回 Cursor。
9. Cursor 把结果交给模型。
10. 模型组织自然语言回答用户。
```

所以，MCP 的重点不是“大模型拿到了数据库连接”，而是“AI 应用用标准方式连接了一个能提供数据库能力的 Server”。

## 4. Tools、Resources、Prompts 怎么区分

对 MCP Server 来说，最核心的三类 primitives 是 tools、resources 和 prompts。

先用一张表建立直觉：

| 类型 | 作用 | 典型例子 | 判断标准 |
| --- | --- | --- | --- |
| Tool | 执行动作 | 执行 SQL、创建 GitHub issue、调用 API | 是否要做一件事 |
| Resource | 提供上下文 | README、数据库 schema、日志、配置文件 | 是否主要是给模型读取资料 |
| Prompt | 复用提问模板 | 日志分析模板、SQL 生成模板、会议纪要提取模板 | 是否主要是在组织任务表达 |

**Tool：执行动作**

Tool 是可以被 AI 应用调用的函数或动作。

例如：

- 执行 SQL 查询
- 创建 GitHub issue
- 调用天气 API
- 发送邮件
- 写入一条任务记录

判断标准很简单：如果它需要执行动作，尤其是会调用外部 API、数据库、文件系统，甚至可能改变外部状态，通常就是 tool。

**Resource：提供上下文**

Resource 是可读取的上下文数据。

例如：

- 项目的 README.md
- 数据库 schema
- 某张表的字段说明
- 一份配置文件
- 一段日志内容

如果它主要是“给模型看的一份资料”，通常就是 resource。

一个容易混淆的例子是读取文件：

```text
某个被 MCP Server 暴露的固定 README.md 文件 = resource
根据用户传入路径读取任意文件 = 可能是 tool
```

这里的 README.md 指被 MCP Server 暴露给 Host 的某个项目文件，不一定是 MCP Server 自己的 README。关键不在于这个文件属于谁，而在于它是作为一份固定上下文被读取，还是通过一次带参数的动作去获取。

固定的、可发现的上下文资料，更适合设计成 resource；带参数执行一次读取动作，则可能被建模成 tool。

**Prompt：复用提问模板**

Prompt 是可复用的消息模板，用来帮助用户或 Host 构造更稳定的模型请求。

例如：

- 请根据这段日志分析可能的错误原因，并按严重程度排序
- 请根据数据库 schema 生成安全 SQL
- 请从会议纪要中提取决策、待办事项和风险点

Prompt 不负责执行外部动作，也不是外部数据本身。它也不会增强模型能力本身，而是把任务表达得更结构化、更可复用。它的重点是“如何问”。

可以先用这句话记住三者区别：

```text
tool = 做一件事
resource = 给一份资料
prompt = 提供一套提问模板
```

## 5. 什么时候需要 MCP

不是所有 AI 应用都需要 MCP。

如果你只是做一个网页聊天机器人，让用户输入普通问题，然后 AI 应用把问题交给模型回答，例如“解释一下什么是递归”，这个场景通常不需要 MCP。

因为它不需要访问外部文件、数据库、API，也不需要执行外部动作。模型和普通应用逻辑就能完成。

MCP 开始有价值的场景，通常有一个共同点：

> AI 应用需要连接外部上下文或外部能力。

例如：

- 读取当前项目文件并总结代码结构
- 查询公司内部数据库
- 检索知识库
- 创建 GitHub 或 Jira issue
- 调用订单系统 API
- 使用设计、浏览器、终端等外部工具

这里也有一个重要边界：

> 不需要额外配置 MCP，不等于大模型自己能直接访问外部数据。

有些 AI 应用已经内置了文件读取、终端、浏览器或代码索引能力。此时用户体感上可能“不需要 MCP”，但底层仍然是 Host 或其他运行时在为模型提供外部上下文和外部能力，而不是模型自己直接访问你的机器。

可以先用一个简单口诀判断：

> 只聊天，不需要；要外部上下文或外部动作，才考虑 MCP。

## 6. 阶段性总结

刚开始学习 MCP，不需要急着写 Server，也不需要马上读完整规范。先把下面几句话记牢：

- MCP 是协议，不是模型、不是工具、也不是 Agent 框架。
- MCP 连接的是 AI 应用和外部系统，不是让大模型直接连接数据库。
- MCP Host 是使用 MCP 的 AI 应用。
- MCP Client 是 Host 内部连接某个 MCP Server 的组件。
- MCP Server 是暴露上下文和能力的程序，通常是外部系统的适配层。
- Tool 用来执行动作，resource 用来提供上下文，prompt 用来复用提问模板。
- 只有当 AI 应用需要外部上下文或外部能力时，MCP 才开始有明显价值。

学到这里，你已经可以判断一篇 MCP 文章有没有把基本概念讲清楚。

下一步再进入 Host、Client、Server 的架构细节，会更自然：你不是在背名词，而是在理解一个 AI 应用如何安全、标准地连接外部世界。
