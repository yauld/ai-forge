# AI Forge 新文章模板

这份模板用于后续新文章。目标是让公众号文章更短、更清晰，同时让 GitHub 保留完整实验、代码、Notebook、截图和运行说明。

## 一句话定位

```text
这篇文章回答的问题：

读者读完后应该获得的能力：

对应 GitHub 实验目录：
```

## 公众号文章结构

```text
# 标题：用具体问题命名

## 1. 为什么这个问题值得讲

用真实开发场景开头，避免先堆概念。

## 2. 先建立一个最小直觉

只放必要图示、关键代码或最小例子。

## 3. 工程实现里真正容易踩坑的地方

写判断、边界、取舍，不写流水账。

## 4. 我的实验结论

总结这次实验真正验证了什么，以及没有验证什么。

## 5. 深入复现入口

完整代码、Notebook、截图和运行说明：
https://github.com/yauld/ai-forge/tree/main/<path>
```

## GitHub 实验页结构

每篇新文章建议配一个稳定实验页，优先使用下面结构：

```text
labs/<topic>/<lesson>/
├── README.md
├── article.md
├── notebook.ipynb
├── assets/
└── data/
```

独立服务、脚本和最小项目放到 `examples/`。文章专属素材放在实验目录的 `assets/`，跨主题复用素材放在顶层 `assets/`。

## 实验页 README 模板

```markdown
# 实验标题

## 解决什么问题

用 2 到 4 句话说明这节实验的目标。

## 适合谁

- 已经读过公众号文章，想复现实验的读者。
- 想直接查看完整代码和 Notebook 的开发者。

## 文件说明

| 文件 | 说明 |
| --- | --- |
| `article.md` | 公众号文章或长文版本 |
| `notebook.ipynb` | 可运行 Notebook |
| `examples/` | 示例脚本或服务 |
| `assets/` | 截图和图示 |
| `data/` | 示例数据 |

## 快速运行

```bash
uv sync
uv run jupyter notebook
```

## 公众号入口

文章发布后再补充链接。
```

## 发布前检查

- 标题是否只回答一个问题。
- 公众号文章是否删除了大段代码、长截图和排障过程。
- GitHub 实验页是否能独立看懂和复现。
- README 是否包含文件说明和最小运行命令。
- 文章结尾是否指向具体 GitHub 目录，而不是只给仓库首页。
- Notebook 是否清理了无关输出、个人路径和敏感信息。
- 示例数据是否可以公开。
