# 11 | MCP 审计安全：日志不是记得越多越好

很多系统把“记录完整日志”当作一种安全习惯。

一次退款请求来了，订单编号、退款原因、用户输入、幂等键全部写进日志。以后出了
问题，确实很好查。

但这份“完整”也意味着：原本只在一次调用中短暂出现的敏感数据，从此进入了保存
更久、访问者更多的日志系统。

日志本来用来调查泄露，最后却可能成为泄露源。

所以，审计真正追求的不是“尽可能完整”，而是：

> 用尽可能少的敏感数据，留下足以还原事实的证据。

## 1. 完整日志为什么可能更危险

假设一个 MCP Server 收到退款 Tool 调用：

```json
{
  "order_id": "O-1005",
  "reason": "duplicate",
  "idempotency_key": "refund-audit0001"
}
```

最省事的做法，是把整个请求原样保存。

可审计系统的生命周期和业务请求并不相同。请求可能只存在几秒，日志却可能保留
几个月；请求只经过一个服务，日志却可能被开发、运维、安全平台和第三方采集系统
读取。

一旦完整请求进入日志，敏感数据就多了一份副本、多了一条传播路径，也多了一批
潜在访问者。

这并不是说日志应该什么都不记，而是不能把“以后也许有用”当作无限收集数据的
理由。审计之前应该先问：

> 调查这类操作，最少需要哪些信息？

## 2. 好的审计，既记录结果，也控制边界

一笔退款至少有两种值得记录的结果：

```text
applied：操作已经执行
denied：操作到达系统，但被拒绝
```

成功记录解释订单为什么发生了变化；拒绝记录则暴露异常尝试。

如果只记录成功，有人连续尝试操作不存在的订单时，调查者看到的仍是一片安静。
如果只记录失败，真实退款发生后，又很难解释是谁改变了业务状态。

但两类记录都不需要保存完整请求。对于退款，通常只需要：

| 应该记录 | 不应直接记录 |
| --- | --- |
| 时间、动作、订单编号 | 完整 Tool 请求 |
| 成功或拒绝结果 | Token、原始幂等键 |
| 必要的金额和业务原因 | 无关的订单与用户信息 |
| 用于关联的短指纹 | “以后可能有用”的全部字段 |

这里的关键不是机械地减少字段，而是让每个字段都能回答一个调查问题：

```text
什么时候发生？
针对哪个对象？
系统执行了还是拒绝了？
为什么得到这个结果？
它与哪一次业务意图有关？
```

回答不了这些问题的日志不够用；与这些问题无关的数据，则不该顺手进入审计。

失败信息也要控制粒度。与其分别告诉调用方“订单不存在”“订单已取消”或“订单
已经退款”，系统可以统一返回“当前不可退款”，避免错误差异反过来帮助外部调用者
枚举业务状态。

审计安全不是少记，而是有目的地记。

## 3. 不保存原始凭据，怎样关联事件

不保存原始幂等键之后，还有一个现实问题：业务响应、退款记录和审计事件怎样对应
起来？

一种常见做法，是在系统内部保存完整摘要用于精确判重，在响应和审计中只保留摘要
的一小段作为短指纹：

```text
原始幂等键
  → 完整摘要：系统内部判重
  → 短指纹：响应与审计之间的人工关联
```

于是，成功退款可以留下：

```text
order_id: O-1005
outcome: applied
amount: 129.0
fingerprint: 55740cd0fe30
```

拒绝请求可以留下：

```text
order_id: O-9999
outcome: denied
reason: order_not_refundable
fingerprint: efa29aa6ecdc
```

原始值没有进入审计，两次尝试却仍然拥有可关联的标识。

短指纹只能用于关联，不能作为凭据，也不能承担唯一约束。摘要同样不是加密：如果
原始值非常容易猜，攻击者仍可能枚举候选值进行比对。因此，摘要和短指纹仍然需要
访问控制与保留期限。

减少暴露面，不等于获得了绝对匿名。

## 4. 日志说成功，不代表业务真的成功

看到一条：

```text
outcome: applied
```

能不能证明退款已经完成？

不能。

它只能证明系统写下了“成功”。如果日志已经写入，订单更新却失败，数据库里就会
出现一条成功记录，订单仍然是 `paid`。

要避免这种错位，业务更新和审计写入应该处在一致的提交边界中：

```text
更新订单
  → 写入幂等记录
  → 写入审计
  → 一起提交或一起回滚
```

调查和测试时，还要回查最终业务状态：

```text
审计结果：applied
订单状态：refunded
```

这两项证据回答的问题不同。

审计记录说明系统如何解释这次操作，最终状态说明副作用是否真的发生。对于退款、
删除、转账或发送消息这类危险操作，不能因为日志写着成功，就跳过业务事实验证。

系统对自己的描述，不等于事实本身。

## 5. 三个问题，检查审计是否真的安全

设计危险操作的审计时，可以先问三个问题：

1. 成功和拒绝是否都留下了足够调查的证据？
2. 日志中是否出现了原始凭据、完整请求或无关数据？
3. 审计结果能否与最终业务状态相互验证？

真实系统还需要保护审计本身：限制谁能读取、规定保存多久、防止记录被篡改，并对
高频拒绝和异常操作建立告警。但这些措施都建立在同一个前提上——先明确什么证据
值得留下，什么敏感数据不该进入日志。
