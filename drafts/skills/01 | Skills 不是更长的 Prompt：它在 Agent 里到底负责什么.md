# 01 | Skills 不是更长的 Prompt：它在 Agent 里到底负责什么

很多人第一次看到 Skills，会下意识把它理解成“更长的 Prompt”。

这个反应很正常。毕竟它看起来也是一份文本说明，也会告诉 Agent 该怎么做，还经常包含步骤、约束、模板和检查清单。

但如果只把 Skill 当成长 Prompt，很快会遇到一个问题：所有东西都会被塞进提示词。

周报格式塞进去，会议纪要规范塞进去，合同初筛流程塞进去，工具使用规则也塞进去。最后得到的不是一个清晰的 Agent 能力系统，而是一团越来越长、越来越难维护的提示词。

Skill 真正解决的问题，不是“怎么写一段更强的提示词”，而是：

> 当 Agent 反复遇到同一类任务时，怎样把这类任务的工作方法沉淀下来。

## 一、Agent 需要的不只是工具

假设我们要做一个 AI 数字运营助理。

它每天可能要整理会议纪要、生成周报、查询制度口径、更新任务系统，还可能要初筛合同或报销材料。

这些事情看起来都可以交给大模型，但真做起来会发现，它们并不是单纯的“问答任务”。

整理会议纪要，需要知道哪些信息必须保留，哪些内容不能乱补。

生成周报，需要知道结构、口径、数据缺失时怎么处理。

查询制度，需要知道什么时候去知识库里找资料。

更新任务系统，需要知道哪些动作可以自动执行，哪些必须人工确认。

合同初筛更复杂：先抽取关键信息，再查风险条款，再生成意见，最后标注需要人工判断的部分。

这里面有模型推理，有知识检索，有工具调用，也有流程编排。

如果只给 Agent 一堆 Tool，它知道“能做什么动作”，但不一定知道“这类任务应该怎么做”。

这就是 Skills 要补的位置。

## 二、Skill 更像工作手册

一个比较容易理解的类比是：Skill 像给新员工的工作手册。

新员工入职后，你不会只告诉他：

```text
公司有文档系统、任务系统、数据库和日历。
```

这当然重要，但还不够。

你还要告诉他：

```text
遇到周报任务时，先收集本周进展，再核对数据口径；
结论不确定时要标注，不要编造；
涉及跨团队事项时要列出待确认人；
最后按固定结构输出。
```

前者是工具清单，后者才是工作方法。

Skill 关注的就是后者。

它描述一类任务什么时候触发，按什么步骤处理，需要读取哪些资料，可以调用哪些工具或脚本，最后怎样检查结果。

这也是它和 Prompt 最大的区别。

Prompt 更像这一次的临时交办：

```text
请把下面的会议记录整理成一份周报。
```

Skill 更像长期沉淀下来的岗位 SOP：周报应该怎么整理，哪些信息必须保留，缺失数据怎么处理，输出前要检查什么。

## 三、把 Skill 放回 Agent 架构里

在一个 Agent 系统里，可以粗略这样理解几个组件的职责：

```text
用户任务
  → Agent Host / Runtime
      ↔ LLM
      → Skills
      → RAG
      → MCP / Tools → 外部系统
      → LangGraph
```

这不是固定调用链，也不是说每次请求都要走完这些环节。

它只是说明职责边界。

LLM 负责理解、推理和生成。

RAG 负责查事实和背景资料，比如制度条款、产品说明、历史案例。

MCP / Tools 负责提供可调用动作，比如查数据库、读文件、创建任务。

LangGraph 适合承载复杂流程，比如状态流转、分支、人工确认和可恢复执行。

Skill 站在这些组件之间，描述一类任务的方法：什么时候该用这套流程，执行时要读哪些资料，哪些工具可以用，最后怎样算完成。

所以 Skill 不是模型，也不是工具，也不是知识库。

它是一层任务方法。

## 四、为什么不用 Prompt 全写完

如果只是一次性任务，Prompt 足够。

比如让模型改一句话、总结一段文本、解释一个概念，直接写清楚当前需求就行。

但一旦任务会反复出现，Prompt 就会开始失控。

每次写周报，都要重复说明格式。

每次做审查，都要重复提醒检查项。

每次调用工具，都要重复强调边界。

团队里多人协作时，不同人写的 Prompt 还可能口径不一致。

Skill 的价值在这里就很明显：把稳定的任务方法从临时 Prompt 里抽出来，变成可复用、可发现、可加载、可检查的能力包。

用户只需要表达当前任务，Agent 再判断是否命中某个 Skill。

命中后，再加载对应的流程说明、参考资料和检查标准。

这比每次都从零写一大段 Prompt 更适合长期维护。

## 五、为什么 Skill 也不是 Tool

Tool 回答的是“能不能执行某个动作”。

比如：

```text
read_document
query_orders
create_calendar_event
send_email
```

这些动作很重要，但它们不会自动告诉 Agent 任务该怎么完成。

以“生成运营周报”为例，工具可能能读取数据、查询任务系统、保存文档。

但它不会回答：

```text
先读数据还是先读会议记录？
周报按项目组织还是按业务线组织？
缺失数据能不能补？
哪些结论必须标注为推断？
输出前要不要人工确认？
```

这些判断标准属于 Skill。

Tool / MCP 提供动作，Skill 规定方法。

即使命中了 Skill，也不代表 Agent 可以随便执行工具。真正调用外部系统时，Host、权限策略、工具实现和 MCP Server 仍然要负责安全边界。

## 六、Skill 和 RAG、LangGraph 怎么配合

Skill 也不应该变成知识库。

比如合同初筛任务里，风险条款清单、法务术语解释、历史案例，都可以放进 RAG 或 references 里按需读取。

Skill 正文只需要保留流程：

```text
先抽取合同关键信息；
再检索风险条款；
再生成初筛意见；
最后标注不确定项和人工确认点。
```

这样 Skill 会更短，路由更清楚，资料也可以独立更新。

如果任务进一步复杂，比如需要多轮状态、分支、人工确认或可恢复执行，就可以交给 LangGraph。

这时 Skill 可以像说明书一样告诉 Agent：什么请求应该进入这张图，每一步关注什么，最终怎样验收。

LangGraph 负责运行时流程，Skill 负责解释什么时候用、用来做什么、怎样算做完。

## 七、一个好记的判断

可以用一句话判断 Skill 的位置：

> Prompt 表达这一次要做什么，Tool / MCP 提供能调用的动作，RAG 提供事实上下文，LangGraph 承载复杂流程，Skill 沉淀一类任务的方法。

如果你发现某段 Prompt 反复出现，里面既有步骤、规则、模板，又有检查清单，那么它很可能应该被整理成 Skill。

如果你发现一个 Tool 描述越来越长，开始解释业务流程、判断标准和写作规范，那么也要警惕：这些内容可能不该塞进 Tool，而应该交给 Skill。

Skills 的价值，不是让模型凭空变聪明，而是把团队已经验证过的工作方法沉淀下来。

让 Agent 遇到相似任务时，不必每次都重新摸索。

```text
GitHub 仓库：
https://github.com/yauld/ai-forge

完整实验文章：
labs/skills/foundations/01 | Skills 的定位：它在 Agent 架构中解决什么问题.md
```
