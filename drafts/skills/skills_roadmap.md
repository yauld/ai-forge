# Skills 系统学习路线图

> **主线定位**：把 Skills 作为一种 Agent 能力组织模式来学习。
>
> 目标是用 `SKILL.md` 描述任务触发条件、执行流程、参考资料和可执行脚本，理解 Skills 如何组织 Agent 能力；必要时可用熟悉的 Python 和本地 Ollama 环境做最小实验实现，并逐步接入 RAG、MCP 与 LangGraph，最终形成一个可复用、可评测、可治理的 AI 数字运营助理。

**可用环境**：熟悉 Python；本地可使用 Ollama（qwen3-coder:30b）。它们是实验实现选项，不限定 Skills 学习路线。
**前置知识**：LangChain、LangGraph、RAG、MCP（已学完）
**产出形式**：每阶段至少 1 篇实验文章；当阶段需要验证运行机制时，再补充 1 份可运行代码。

---

## 贯穿案例：养成一个 AI 数字运营助理

把整个学习过程比作带一个新员工从入职到独当一面：

- **Skills** = 工作手册 / SOP，告诉 Agent 遇到某类任务时该按什么方法做。
- **RAG** = 公司知识库 / 制度文档，Agent 遇到不确定事实时去查。
- **MCP** = 办公系统权限和工具接口，Agent 用它连接文件、数据库、邮件、日历等外部系统。
- **LangGraph** = 工作流程图，让复杂任务按状态、分支和人工确认一步步推进。

这个案例从简单办公任务开始，逐步扩展到会议纪要、周报生成、文档处理、合同初筛、知识库查询、MCP 工具调用和审批流程编排。每个阶段就是教这个数字运营助理学会一项新能力。

## 本系列先回答什么问题

1. **Skills 到底放在 Agent 架构的哪一层？**
   - 它不是模型本身，不是 RAG 知识片段，也不是 MCP 工具。
   - 它更像一层“能力包”：告诉 Agent 什么时候该使用某套流程，以及这套流程如何调用知识、脚本和工具。

2. **为什么不用普通 Prompt 就够了？**
   - Prompt 适合一次性指令。
   - Skill 适合沉淀可复用、可发现、可加载、可评测的任务流程。

3. **一个 Skill 应该包含哪些内容？**
   - 触发条件：什么任务应该使用它。
   - 核心流程：任务应该按什么步骤执行。
   - 参考资料：长规范、模板、示例放在哪里。
   - 可执行能力：什么时候调用脚本、工具或 MCP Server。

4. **Skills 如何和现有 Agent 工程组件协同？**
   - 与 `AGENTS.md` 类似，Skill 可以沉淀长期规则，但粒度更聚焦于某一类任务。
   - 与 Plugin 类似，Skill 可以被打包和共享，但它本身更关注工作流和上下文。
   - 与 MCP、RAG、LangGraph 协同时，Skill 负责描述“何时用、按什么顺序用、结果如何验收”。

## 阶段 3：Reference Skill 与 Action Skill

**员工比喻**：区分“查资料用的手册”和“照着执行的流程卡”。

**学习目标**

**产出物**

**预计用时**：1 天

## 阶段 4：渐进式披露：metadata、正文、资源三层加载

**员工比喻**：新员工先看手册目录，确认相关后再读正文，最后才查附录。

**学习目标**

- 理解渐进式披露如何节省上下文。
- 明确三层加载分别解决什么问题。
- 把 Skills 和 RAG 的相似点、不同点讲清楚。

**核心概念**

- 第一层：只加载所有 Skill 的 `name` + `description`。
- 第二层：命中后读取对应 `SKILL.md` 正文。
- 第三层：执行过程中按需读取 `references/` 或调用 `scripts/`。
- RAG 检索的是知识片段；Skills 加载的是任务流程。

**实验内容**

- 手动模拟 10 个 Skill 的三层加载过程。
- 对比“全量塞进 prompt”和“按需加载”的 token 消耗。
- 记录一次误触发案例，并分析是 description 写得不清楚，还是路由策略不合适。

**产出物**

- 三层加载演算笔记。
- 一张渐进式披露流程图。

**预计用时**：0.5–1 天

## 阶段 5：设计教学用 Skills runtime 的最小架构

**员工比喻**：给新员工搭一个工作台：手册库、调度员、工具箱和操作记录都要有位置。

**学习目标**

- 明确本系列用于实验验证的 runtime 边界。
- 设计最小模块：扫描、路由、加载、执行、记录。
- 如需本地模型参与，验证 Ollama + qwen3-coder:30b 是否适合结构化决策或 tool calling。

**核心概念**

- Skill Registry：保存本地 Skill 元数据。
- Router：根据用户请求选择候选 Skill。
- Loader：读取 `SKILL.md`、`references/` 和脚本描述。
- Executor：执行模型建议的工具或脚本。
- Trace：记录命中的 Skill、加载的资源和执行结果。

**实验内容**

- 画出本地 runtime 架构图。
- 如需本地模型参与，用 Ollama 跑通一次基础对话。
- 如果选择使用 Ollama，测试 qwen3-coder:30b 的结构化输出或工具调用表现；如果不稳定，准备 JSON 决策 fallback。

**产出物**

- `stage5_architecture.md`：教学用 runtime 架构说明。
- `stage5_model_probe.py`：可选的本地模型基础能力探测脚本。

**预计用时**：1 天

## 阶段 14：Skills + MCP：用 Skill 指导外部工具调用

**员工比喻**：员工拿到公司系统权限，但必须按手册规定的顺序和边界使用。

**学习目标**

- 理解 Skill 与 MCP 的协同方式。
- 用 Skill 指导模型调用 MCP 工具。
- 明确工具权限、参数边界和人工确认点。

**核心概念**

- MCP 是外部能力接口；Skill 是使用这些接口的操作手册。
- Action Skill 可以规定工具调用顺序、参数来源和结果处理方式。
- 危险动作必须经过 runtime 或 human-in-the-loop 约束。

**实验内容**

- 复用 MCP 专题中的文件系统或数据库 MCP Server。
- 编写一个 Skill：查询数据 → 生成报告 → 写入文件。
- 增加一个危险操作案例，例如覆盖已有文件前必须确认。

**产出物**

- 1 个 “Skill 指导 MCP 调用”案例。
- `stage14_mcp_skill_demo.py`。
- 权限边界说明。

**预计用时**：2 天

## 阶段 15：Skills + LangGraph：状态、分支与人工确认

**员工比喻**：员工升级为项目负责人，复杂任务按流程图推进，关键动作会停下来请你确认。

**学习目标**

- 把 Skill 路由、加载和执行放入 LangGraph 状态图。
- 引入状态、分支、checkpoint 和 human-in-the-loop。
- 理解什么时候该用简单编排器，什么时候该用 LangGraph。

**核心概念**

- 路由节点：选择候选 Skill。
- 加载节点：读取正文和 references。
- 执行节点：调用脚本或 MCP 工具。
- 确认节点：发送邮件、写入文件、覆盖数据前暂停。
- Checkpoint：保存中间状态，支持恢复。

**实验内容**

- 将阶段 13 的组合编排和阶段 14 的 MCP 工具调用改造成 LangGraph 状态图。
- 至少包含路由、加载、执行、人工确认 4 类节点。
- 用一次“生成报告并确认写入”的任务验证流程。

**产出物**

- `stage15_langgraph_skills.py`。
- 状态图结构示意图。
- human-in-the-loop 运行记录。

**预计用时**：2.5–3 天

## 阶段 16：Skill 评测：触发率、误触发与输出验收

**员工比喻**：不能只说员工“好像会了”，要用题库检查他什么时候该翻哪本手册，结果是否合格。

**学习目标**

- 为 Skill 建立可回归的评测集。
- 测试触发、拒触发、改写鲁棒性和输出质量。
- 让 description 和正文修改有可观察反馈。

**核心概念**

- 正例：应该触发某 Skill 的请求。
- 反例：不应该触发该 Skill 的请求。
- 改写样本：同一意图的不同表达。
- 输出验收：格式、字段、约束、关键内容是否满足要求。

**实验内容**

- 为 3–5 个 Skill 编写触发测试用例。
- 实现一个最小 eval runner。
- 修改某个 Skill 的 description，观察触发结果如何变化。

**产出物**

- `evals/skills_eval_cases.json`。
- `stage16_eval_runner.py`。
- 一份评测报告。

**预计用时**：1.5–2 天

## 阶段 17：Skill 工程治理：版本、打包、日志与安全审查

**员工比喻**：员工手册正式纳入公司制度管理，谁改了什么、怎么发布、是否安全都要留痕。

**学习目标**

- 建立 Skill 生命周期管理方法。
- 处理版本、打包、分发、日志和安全审查。
- 明确团队共享 Skill 的风险边界。

**核心概念**

- 版本管理：记录 description、正文、references、scripts 的变化。
- 打包分发：压缩 Skill 文件夹，迁移到其他项目。
- 运行日志：记录命中、加载、脚本调用、MCP 调用、拒绝原因。
- 安全审查：来源、脚本、路径、网络请求、危险工具、敏感数据。

**实验内容**

- 为已有 Skill 增加版本字段或变更记录。
- 编写 Skill 打包脚本。
- 用安全清单审查一个脚本型 Skill。
- 设计 3 个对抗输入：路径穿越、未知脚本、诱导危险操作。

**产出物**

- `stage17_pack_skill.py`。
- `security_checklist.md`。
- 一份 Skill 审查报告。

**预计用时**：2 天

## 阶段 18：综合实战：AI 数字运营助理上线

**员工比喻**：员工转正，能按公司制度处理端到端事务，但关键风险动作仍会请你确认。

**学习目标**

- 把前 17 个阶段的能力整合成一个完整闭环系统。
- 验证 Skills runtime 与 RAG、MCP、LangGraph 的协同。
- 形成可演示、可复现、可扩展的综合实验。

**核心概念**

- Skills runtime：扫描、路由、加载、执行、评测、日志。
- RAG：查询公司制度和历史资料。
- MCP：访问外部文件、数据库或业务系统。
- LangGraph：负责端到端流程和人工确认。

**实验内容**

设计并实现一个端到端案例：

> “帮我处理一份合同 PDF：提取关键条款 → 结合公司知识库判断是否存在风险条款 → 查询历史合同或审批记录 → 按公司格式生成初筛报告 → 人工确认后写入指定位置或准备邮件草稿。”

注意：该案例定位为“内部流程初筛”，不输出法律结论。

该案例需要用到：

- 阶段 10 的脚本型 Skill。
- 阶段 12 的语义路由。
- 阶段 14 的 Skills + MCP 协同。
- 阶段 15 的 LangGraph 流程编排。
- 阶段 16 的评测。
- 阶段 17 的日志与安全审查。

**产出物**

- 一套完整可运行的教学版迷你 Skills runtime。
- 一篇总结性实验文章。
- 一份后续扩展计划：更多 MCP Server、更多 Skill 库、团队共享与权限治理。

**预计用时**：3–4 天

## 长期注意事项

- 本路线图学习的是 Skills 作为 Agent 能力组织模式的设计思想；Python + Ollama 是可用实验环境，不等同于路线图的强制技术栈。
- 教学用 runtime 是为了拆解和验证扫描、路由、加载、执行、评测、治理等机制，不要把本地实现误认为官方唯一规范。
- 每个阶段都要保留最小可复现证据：输入、命令、关键输出、失败样例、结论和边界。
- `name` 与 `description` 是 Skill 发现和路由的核心接口，每次修改都要同步更新触发正例、反例和误触发记录。
- 不要把 Skill 写成大 Prompt：Skill 描述任务流程，RAG 管事实知识，MCP / tools 管外部动作，LangGraph 管复杂状态流。
- 安全约束从早期阶段就开始验证，包括路径边界、脚本白名单、参数校验、危险动作确认和敏感数据处理。
- 每个阶段只回答一个核心研究问题，避免提前混入后续阶段能力。
- 失败样例要沉淀为评测资产，用于后续回归测试和 description 调优。

## 学习建议

1. **先把定位立住**：本系列学习的是 Agent 能力工程化，重点是把可复用任务流程沉淀成 Skill。
2. **阶段 1–4 先吃透概念**：尤其是 Skills 与 Prompt、RAG、MCP、LangGraph 的职责边界。
3. **阶段 5–10 先实现教学用最小 runtime**：不要一开始就追求通用框架，先把扫描、路由、加载、执行跑通。
4. **阶段 11–13 解决规模问题**：Skill 多起来以后，重点观察路由误判、组合调用和中间结果传递。
5. **阶段 14–15 融合旧知识**：MCP 负责外部动作，LangGraph 负责复杂流程，Skill 负责告诉 Agent 该怎么用它们。
6. **阶段 16–17 不要省略**：没有评测和治理的 Skill 只是提示词文件，有了回归测试、安全审查和日志，才像一个可维护的软件工件。
7. **每个阶段结束后都写实验文章**：建议按“研究问题 → 核心概念 → 实验设计 → 运行证据 → 边界与下一步”的结构整理。
